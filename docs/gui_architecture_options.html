Two Operating Modes for MBE Growth

The project supports two modes of operation, switchable depending on whether a human scientist or the AI system is driving the growth:

Normal Mode
Human-Driven Growth
Standard lab hardware (integrated PSU, not easily programmable). RHEED camera and pyrometer display on screen as usual via kSA 400 and TemperaSure. Human scientist watches patterns, makes judgment calls, manually adjusts temperature. A few key RHEED images are noted throughout the growth — mostly experience-driven. Our GUI is not required in this mode. 
RHEED --> kSA screen     (human watches)
Pyro  --> TemperaSure    (human reads)
PSU   --> manual knobs   (human adjusts)
                         "I hope this works out"
AI-Scientist Mode
Closed-Loop Autonomous Growth
Our System
OWON PSU swapped in (programmable, serial-controllable). Same RHEED camera and pyrometer, but data is continuously streamed to our ML pipeline. Classifier2 identifies the current surface reconstruction every frame. RL policy decides temperature adjustments. PID loop drives the OWON. Human monitors via the Growth Monitor GUI and can override at any time. 
RHEED --> ScreenGrab --> Classifier2 --> RL Policy
Pyro  --> Modbus RTU --->---|                 |
                              v                 v
                         Growth Monitor GUI    PID --> OWON PSU
                         (human monitors,          |
                          can override)          v
                                          Substrate heater

Current Status

The Growth Monitor GUI is built and functional with dummy drivers. Classifier2 is trained (100% ideal-split accuracy with full 159-image reference set across all 4 classes). The RHEED gun is down and pyrometer was off during our last lab visit, so no live data testing has been possible yet.

The core question: where does the GUI + ML pipeline run, and how does it connect to hardware? In AI-Scientist mode, the system must have direct access to the OWON PSU (USB serial), pyrometer (COM4), and RHEED camera (kSA window or GigE) — all on the same machine. This constrains the architecture.

The Four Options

Option A
Screen-Grab Over TeamViewer
GUI runs on your Mac. TeamViewer shows Bulbasaur's desktop. The Mac captures its own screen (which shows the TeamViewer window showing the kSA window) using mss. Classifier2 runs locally on MPS. Pyrometer data scraped from the TemperaSure window visible through TeamViewer. 
kSA window (Bulbasaur) --TeamViewer--> Mac screen --mss--> ScreenGrabCamera
TemperaSure (Bulbasaur) --TeamViewer--> Mac screen --OCR?--> Pyrometer reading
                                                       ^^ double-compressed, lossy

Pros

  • Zero install on Bulbasaur
  • Works today (TeamViewer creds confirmed)
  • ML runs on Mac MPS GPU

Cons

  • Double-compressed frames (TeamViewer encoding)
  • Image quality degraded vs training data
  • Manual window positioning required
  • Pyrometer needs OCR or manual read
  • High latency (~1-3s round trip)

Requirements

  • TeamViewer session active
  • kSA window positioned consistently
  • Stable campus network
Option B
Install Python on Bulbasaur, Run GUI Locally
Recommended
Install Python + our AIQM code directly on Bulbasaur. ScreenGrabCamera._grab_win32() captures the kSA window natively at full resolution with the exact false-color LUT that Classifier2 was trained on. Pyrometer reads via COM4 Modbus RTU (driver already written). When the MBE power supply replaces the OWON, its control also runs here. This is the production architecture. 
kSA window --win32gui+mss--> ScreenGrabCamera  (native, lossless)
COM4 --Modbus RTU--> ModbusPyrometer     (direct serial, no scraping)
MBE PSU --serial/GPIB--> PowerSupplyWorker  (future, same PC)
         |
         v
   Growth Monitor GUI  -->  Classifier2 (CPU)  -->  Auto-capture + Logging

Pros

  • Best image quality (matches training data)
  • Direct serial to pyrometer (Modbus RTU)
  • Lowest latency (all local)
  • No network dependency
  • Production-ready for MBE PSU integration
  • Simplest architecture

Cons

  • Need permission to install Python
  • No MPS/CUDA GPU (CPU inference only)
  • Must maintain code on lab PC

Requirements

  • Python 3.10+ on Bulbasaur
  • pip install PyQt6, torch (cpu), etc.
  • Admin or user-level install permission
  • ~350 MB disk (Python + PyTorch CPU-only + deps)
Option C
Frame Server on Bulbasaur, GUI on Mac
Lightweight Python script on Bulbasaur captures kSA frames + pyrometer serial, streams them over ZMQ or HTTP to your Mac. GUI and Classifier2 run on Mac with MPS acceleration. Hybrid approach: minimal install on lab PC, heavy compute on Mac. 
kSA window --win32gui--> frame_server.py --ZMQ/HTTP--> Mac GUI
COM4 --Modbus--> frame_server.py --ZMQ/HTTP--> Mac GUI
                                  ^^ campus network required

Pros

  • Good image quality (native capture)
  • ML runs on Mac MPS GPU
  • Minimal install on Bulbasaur

Cons

  • Still needs Python on Bulbasaur
  • Network latency + firewall issues
  • Two machines to maintain
  • Doesn't scale to MBE PSU control

Requirements

  • Python on Bulbasaur (minimal)
  • Network port open between machines
  • ZMQ or HTTP streaming code
Option D
Direct GigE Camera Stream via SENA Device
Bypass kSA entirely. Tap into the Allied Vision camera's GigE stream through the SENA serial-to-Ethernet bridge at 10.0.42.231, or directly via the instrument subnet. Requires understanding the SENA device's role and possibly reconfiguring the camera's GigE stream to allow multiple consumers. 
Manta G-033B --GigE--> instrument subnet --> vmbpy direct capture
                     ^^ conflicts with kSA 400 (exclusive lock)
SENA 10.0.42.231 --SSH--> ???  (purpose unknown, needs investigation)

Pros

  • Raw 12-bit monochrome data
  • No screen scraping
  • Highest possible frame rate

Cons

  • Loses false-color LUT (Classifier2 mismatch)
  • May conflict with kSA exclusive camera lock
  • SENA device purpose unknown
  • Requires significant R&D

Requirements

  • SSH into SENA to discover its role
  • Allied Vision SDK (vmbpy) on capture machine
  • Retrain Classifier2 on raw frames, or apply LUT

Recommendation: Option B — Build for the Lab PC

The AI-Scientist mode requires direct access to three hardware interfaces on the same machine: OWON PSU (USB serial for PID control), pyrometer (COM4 Modbus RTU), and RHEED camera (kSA screen grab). This rules out any remote-only architecture. The GUI must run on Bulbasaur.

The development workflow becomes: develop and train on Mac, deploy and run on Bulbasaur. Model training uses MPS GPU acceleration on the Mac. Trained artifacts (best_model.pth) are copied to Bulbasaur where inference runs on CPU at ~80 ms per frame. Full closed-loop budget is ~120 ms/cycle (screen grab 10 ms + Modbus read 20 ms + inference 80 ms + RL 1 ms + PID + serial write 11 ms) — well within the 1 FPS target.

The only prerequisite is permission to install Python on the lab PC. This is the most easily actionable blocker of all the options.

01 Get permission to install Python on Bulbasaur — user-level install via winget, Microsoft Store, or embeddable zip (no admin needed)
02 Clone AIQM repo + copy model artifacts — via git (if installable) or USB drive. Include best_model.pth and ideal reference images
03 pip install dependencies — PyQt6, torch (CPU-only wheel, ~150 MB), pymodbus, mss, pillow, numpy
04 Connect OWON PSU via USB — verify serial port detection and PID loop on Bulbasaur
05 Test pyrometer serial — Modbus RTU read on COM4 when pyrometer is powered on
06 Test kSA screen grab — verify ScreenGrabCamera._grab_win32() captures the RHEED window natively
07 Full integration test — Growth Monitor GUI with live RHEED + pyrometer + Classifier2 + OWON PID + auto-capture logging
